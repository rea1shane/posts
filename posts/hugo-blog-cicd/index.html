我的博客是使用 Hugo 搭建的，并将其部署在 GitHub Pages 上。与官方教程不一样的是，我把完整的博客项目拆分成了两个仓库，分别管理博客的框架与内容，达到了一个仓库专注于一件事情的目的。但是这样就需要两个仓库合起来才能成为一个完整的博客项目，这篇文章就来分享一下我是如何在这个场景下完成 CI/CD 的。

在进入正题之前，先来介绍一下两个仓库的内容。标准 Hugo 博客项目的基础结构如下：

blog/
├── assets/
├── content/
│   └── posts/ # 博文内容
├── layouts/
├── static/
├── themes/
└── hugo.yaml

其中 content/posts 目录用于存放博文，其他的文件则负责博客的样式，所以我将 posts 部分抽取出来单独作为博文仓库，然后通过 Git 的 submodule 功能将其作为作为博客框架仓库（以下简称框架仓库）的一个子模块：

git submodule add --depth=1 https://github.com/rea1shane/posts.git content/posts

这样两个仓库合为一体的同时也保持了相互独立。

现在我可以选择将博客部署在其中一个仓库的 GitHub Page 上，但是因为我私有化了框架仓库，GitHub Pages 又不允许免费用户在私有仓库上部署网站，所以我只能选择将博客部署在博文仓库上。部署在博文仓库也是有好处的，如果以后更换了博客框架，可以很简单的将博客的 CI/CD 流程迁移到新的项目上，而博文仓库几乎不需要进行任何变动，而且博文才是博客的核心，部署在这里也算是合情合理。

将博客部署到 GitHub Pages

如果博客项目是一个整体的话，只需要一个简单的 GitHub Action 就可以完成部署流程：

# https://github.com/marketplace/actions/hugo-setup#getting-started

name: GitHub Pages

on:
  push:
    branches:
      - main  # Set a branch to deploy
  pull_request:

jobs:
  deploy:
    runs-on: ubuntu-22.04
    concurrency:
      group: ${{ github.workflow }}-${{ github.ref }}
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: true  # Fetch Hugo themes (true OR recursive)
          fetch-depth: 0    # Fetch all history for .GitInfo and .Lastmod

      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v2
        with:
          hugo-version: '0.119.0'
          # extended: true

      - name: Build
        run: hugo --minify

      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        if: github.ref == 'refs/heads/main'
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./public

但是现在博客的框架与博文独立开了，需要在框架仓库中运行这个 GitHub Action，将构建后的内容部署在博文仓库中，那么就需要对 Action 进行一点小小的改动。

负责部署的 Action 是 peaceiris/actions-gh-pages ，它的 external_repository 参数支持将内容部署在非运行该 Action 的仓库中：

- name: Deploy
  uses: peaceiris/actions-gh-pages@v3
  with:
    deploy_key: ${{ secrets.ACTIONS_DEPLOY_KEY }}
    external_repository: username/external-repository
    publish_branch: your-branch
    publish_dir: ./public

这一行为需要目标仓库的写入权限，GITHUB_TOKEN 只有对本仓库的权限，我们就需要创建一个 PAT 来让框架仓库有对博文的写入权限。

创建用于部署代码的 PAT

可以直接访问 https://github.com/settings/tokens?type=beta 进入或按照以下步骤进入 GitHub 的 PAT 管理页面 1：

  1. 进入 GitHub 网页。
  2. 点击右上角的头像。
  3. 点击 Settings。
  4. 点击左侧栏最下方的 Developer settings。
  5. 展开 Personal access tokens，点击 Fine-grained tokens。

点击 Generate new token 按钮创建一个新的 PAT，需要进行的设置如下：

  • Token name：为 token 起一个名字。
  • Expiration：Token 的过期时间。最长设置为一年 2。Token 过期后可以通过其详情页中的 Regenerate token 按钮重新生成 3。
  • Description：描述这个 token 的作用。
  • Resource owner：选择这个 token 是使用哪个用户创建的，选项中还会包含你所在的组织。因为博客仓库是管理在自己账号下的，所以选自己即可。
  • Repository access：选择仓库的可访问性。这里我建议权限最小化原则，因为我们是要将博客代码部署到 posts 仓库中，所以选择 Only select repositories 选项，然后只添加 posts 仓库即可。

接下来就改设置 token 所具有的权限，因为部署 GitHub Page 实际上是向仓库的分支提交构建后的网页代码，所以需要拥有对仓库代码的写权限。展开 Permissions 中的 Repository permissions，将 Contents 部分的权限修改为 Read and write。进行完该设置后，Metadata 部分会被强制设置为 Read-only 的访问权限，并且有 mandatory 的提示。

在提交之前，可以在按钮的上方看到所有设置的概览，它应当与这个类似：

确认提交之后，你有且仅有一次机会复制 token 的内容！刷新页面后就没有机会再找回 token 内容了。如果你因为各种原因丢失了 token 内容，可以通过 Regenerate token 按钮再生成一次。

提示：如果觉得创建 PAT 的流程不够清晰，或者上述内容已经过时，可以查看 官方文档 获取更多细节。

将 PAT 添加到框架仓库的 secrets 中

在完成 PAT 的创建后，我们还需要把 token 记录到框架仓库的 secrets 中，使构建 Action 可以使用该 token，进而能够将构建后的网站代码推送到博文仓库。

可以通过访问 https://github.com/username/repository_name/settings/secrets/actions 4 或按照以下步骤进入框架仓库的 secrets 管理页面：

  1. 进入框架仓库的 GitHub 页面。
  2. 点击 Settings 标签卡。
  3. 展开左侧栏中的 Secrets and variables，点击 Actions。

然后按照以下步骤创建一个 secret：

  1. 点击 New repository secret 按钮。
  2. 在 Name 中输入一个名称 5。
  3. 在 Secret 中输入刚刚创建的 token 内容。
  4. 点击 Add secret 按钮完成创建。

修改部署 Action

现在我们需要配置部署 Action 使用该 token，将原先 Deploy 步骤的 github_token 更换为 personal_token，并且指定外部仓库，这是一个我的例子：

  - name: Deploy
    uses: peaceiris/actions-gh-pages@v3
    if: github.ref == 'refs/heads/main'
    with:
-     github_token: ${{ secrets.GITHUB_TOKEN }}
+     personal_token: ${{ secrets.GH_PAGES_TOKEN }}
+     external_repository: rea1shane/posts
      publish_dir: ./public

你需要将 personal_token 和 external_repository 的内容更换为自己的。

在所有内容都正确的设置后，尝试向框架仓库 push 一些内容，你会发现网站代码被推送到了博文仓库的 gh-pages 分支 6，这就说明大功告成啦，剩下的步骤就可以像官方文档一样，设置自己的自定义域名啦。

更多的的自动化

我们在搭建好网站后，一般只会修改博文仓库，并且希望在更新完博文仓库后博客可以自动的更新部署。但是现在存在一个问题：博文仓库和框架仓库是通过 submodule 联系在一起的，框架仓库没有办法自动感知到博文的更新，博客的 CI/CD 又是由框架仓库的动作触发和完成的，这就导致我们在更新完博文仓库后什么也不会发生。

接下来我们来再做一些事情，让流程更加符合我们的期望。

自动更新子模块

首先我们写一个 Action 来让框架仓库更新子模块：

name: Update Posts

on:
  workflow_dispatch:

jobs:
  update:
    name: Update Posts
    runs-on: ubuntu-latest
    steps:
      - name: Checkout parent
        uses: actions/checkout@v4

      - name: Checkout posts
        uses: actions/checkout@v4
        with:
          repository: rea1shane/posts
          path: content/posts

      - name: Commit
        run: |
          git config --local user.name "github-actions[bot]"
          git config --local user.email "github-actions[bot]@users.noreply.github.com"
          git commit -am "Auto updated posts" || echo "No changes to commit"          

      - name: Push
        uses: ad-m/github-push-action@master
        with:
          branch: ${{ github.ref }}

这个 Action 会更新 posts 这个 submodule，并且使用 github-actions[bot] 这个机器人推送到触发 Action 的分支。

这个 Action 的触发条件是 workflow_dispatch ，这是一种手动触发的事件，你可以使用 GitHub API 、 GitHub CLI 或 GitHub web 界面手动触发工作流的运行，并且这种事件只能在默认分支上被触发。

现在我们需要在博文仓库更新后，触发框架仓库的这个工作流。我们立马想到了在博文仓库中添加一个 Action，使其在更新后触发框架仓库的工作流，这里就需要使用 GitHub API 来 创建工作流调度事件 ，模板代码如下：

curl -L \
  -X POST \
  -H "Accept: application/vnd.github+json" \
  -H "Authorization: Bearer TOKEN" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  https://api.github.com/repos/OWNER/REPO/actions/workflows/WORKFLOW_ID/dispatches \
  -d '{"ref":"BRANCH","inputs":{"name":"Mona the Octocat","home":"San Francisco, CA"}}'

它需要我们提供以下信息：

  • TOKEN：因为我们要通过 API 触发一个工作流的调度，所以需要一个令牌进行身份的验证。
  • OWNER/REPO：被触发工作流所位于的仓库。
  • BRANCH：被触发工作流所在的分支名称。这个只能是仓库的默认分支，所以填写默认分支名称即可。
  • WORKFLOW_ID：被触发工作流的 ID。可以将其设置为工作流文件名。例如，我将其设置为 update-posts.yml。

TOKEN 参数可以参考 创建用于部署代码的 PAT 部分创建一个 PAT。核心设置的差异为：

  1. 这个 PAT 用于触发框架仓库的工作流，所以 Repository access 部分还是选择 Only select repositories，添加框架仓库即可。
  2. Permissions 部分同样修改的是 Repository permissions，将 Actions 部分的权限修改为 Read and write。进行完该设置后，Metadata 部分也会被强制设置为 Read-only 的访问权限，并且也会有 mandatory 的提示。

提交按钮上方的概览应当与这个类似：

创建完 PAT 后，需要将其添加到博文仓库的 secrets 中。

现在我们要讲这个 API 包装为一个工作流，将其存放在博文仓库，这个工作流在博文仓库内容变更后触发，继而触发框架仓库的同步博文工作流。Action 内容如下：

name: Notify Website

on:
  push:
    branches:
      - main

jobs:
  notify:
    name: Notify Website
    runs-on: ubuntu-latest

    steps:
      - name: Github REST API Call
        env:
          ACTIONS_TRIGGER_TOKEN: ${{ secrets.WORKFLOW_TRIGGER_TOKEN }}
          WEBSITE_REPO: rea1shane/website
          WEBSITE_BRANCH: main
          WORKFLOW_ID: update-posts.yml
        run: |
          curl \
            -fL \
            --retry 3 \
            -X POST \
            -H "Accept: application/vnd.github+json" \
            -H "Authorization: token ${{ env.ACTIONS_TRIGGER_TOKEN }}" \
            https://api.github.com/repos/${{ env.WEBSITE_REPO }}/actions/workflows/${{ env.WORKFLOW_ID }}/dispatches \
            -d '{"ref":"${{ env.WEBSITE_BRANCH }}"}'

让 Action push 触发 push trigger

最后

总结一下本次博客项目的改造过程：

  1. 拆分：项目拆分为了两个仓库，分别负责框架部分与博文部分，并通过 submodule 联系到了一起。
  2. GitHub Actions（共 3 个）：
    • 框架仓库（2 个）：创建了两个工作流，分别负责更新博文和部署 GitHub Pages。
    • 博文仓库（1 个）：创建了一个工作流，负责触发框架仓库的更新博文工作流。
  3. GitHub PAT（共 3 个）：
    • 框架仓库（2 个）：更新博文工作流使用一个 PAT 用于更新博文仓库的代码，该 PAT 具有博文仓库的读写权限；部署 GItHub Pages 工作流使用一个 PAT 用于推送代码到博文仓库的 gh-pages 分支，该 PAT 拥有博文仓库的读写权限。
    • 博文仓库（1 个）：使用一个 PAT 用于触发博文仓库的工作流，该 PAT 具有博文仓库的 Actions 读写权限。

部署 GItHub Pages 工作流 YAML 文件样例如下：

# gh-pages.yaml 位于框架仓库

name: GitHub Pages

on:
  push:
    branches:
      - main
    paths-ignore:
      - .github/workflows/update-posts.yaml

jobs:
  deploy:
    runs-on: ubuntu-22.04
    concurrency:
      group: ${{ github.workflow }}-${{ github.ref }}
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive

      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v2
        with:
          hugo-version: latest

      - name: Build
        run: hugo --minify

      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        if: github.ref == 'refs/heads/main'
        with:
          personal_token: ${{ secrets.POSTS_GIT_TOKEN }}
          external_repository: rea1shane/posts
          publish_dir: ./public

更新博文工作流 YAML 文件样例如下：

# update-posts.yaml 位于框架仓库

name: Update posts

on:
  workflow_dispatch:

jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout website
        uses: actions/checkout@v4
        with:
          persist-credentials: false

      - name: Checkout posts
        uses: actions/checkout@v4
        with:
          repository: rea1shane/posts
          path: content/posts

      - name: Commit
        run: |
          git config --local user.name "github-actions[bot]"
          git config --local user.email "github-actions[bot]@users.noreply.github.com"
          git commit -am "Auto updated posts" || echo "No changes to commit"          

      - name: Push
        uses: ad-m/github-push-action@master
        with:
          github_token: ${{ secrets.WEBSITE_GIT_TOKEN }}
          branch: ${{ github.ref }}

触发更新博文工作流的工作流 YAML 文件样例如下：

# dispatch-website-workflow.yaml 位于博文仓库

name: Dispatch website workflow

on:
  push:
    branches:
      - main
    paths-ignore:
      - .github

jobs:
  dispatch:
    runs-on: ubuntu-latest
    steps:
      - name: Call GitHub REST API
        env:
          TOKEN: ${{ secrets.WEBSITE_GH_ACTIONS_TOKEN }}
          REPO: rea1shane/website
          REF: main
          WORKFLOW_ID: update-posts.yaml
        run: |
          curl \
            -fL \
            --retry 3 \
            -X POST \
            -H "Accept: application/vnd.github+json" \
            -H "Authorization: token ${{ env.TOKEN }}" \
            https://api.github.com/repos/${{ env.REPO }}/actions/workflows/${{ env.WORKFLOW_ID }}/dispatches \
            -d '{"ref":"${{ env.REF }}"}'
